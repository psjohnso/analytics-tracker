Getting Started

Logging In

The Analytics Project Tracker uses ArcGIS Online OAuth 2.0 authentication. When you first open the application, you'll be redirected to the City of Tucson ArcGIS Online portal to sign in with your organizational credentials.

After successful authentication, your name appears in the upper-right corner. Your session lasts 120 minutes, after which you'll be prompted to re-authenticate.

User Roles

  • Standard User: Can view all tabs (Overview, Projects, Tasks, Resources, Forecast), create and edit projects and tasks, edit allocation percentages, and submit project ideas.
  • Admin: Members of the designated Team Leads group gain access to the Settings tab and can promote project ideas to active status. An Admin badge appears next to your name.

Browser Requirements

The application runs entirely in the browser as a single HTML page. It works with modern versions of Chrome, Edge, Firefox, and Safari. No software installation is required.

Navigating the Application

Tab Bar

The top navigation bar provides access to the main sections:

  • Overview: Dashboard with KPI summary cards showing project counts by status, active tasks, and team allocation overview.
  • Projects: Full list of analytics projects with filtering, sorting, and detail views. Supports grid and list layouts.
  • Tasks: Individual task management with status tracking, due dates, and assignee information.
  • Resources: Team capacity planning with per-person allocation charts, utilization metrics, and an interactive allocation editor.
  • 📊 Forecast: Forward-looking capacity projection showing team availability and utilization trends.
  • ⚙️ Settings: (Admin only) Team member management and absence/time-off editing.

Sidebar Filters

The left sidebar provides filtering options that change based on the active tab. Filter by project status, priority, category, or team member. Active filters appear as highlighted chips — click to toggle. The content area updates immediately.

View Toggle

On the Projects and Tasks tabs, switch between Grid (card-based) and List (table-based) views using the toggle buttons in the toolbar. Grid is the default.

Toolbar Actions

  • 🔄 Refresh: Reloads all data from ArcGIS Online, including projects, tasks, and resource allocations.
  • 💡 Submit Idea: Opens a simplified form for submitting new project ideas. Ideas enter with "Idea" status and must be promoted by an Admin.
  • 📋 Weekly Review: Filters tasks to show only those with activity in the current week.

Overview Tab

The Overview tab provides a high-level dashboard of the team's current workload. It displays summary KPI cards showing the number of projects by status (Active, On Hold, Complete, etc.), task counts, and overall team utilization. This tab hides the sidebar and view toggle to provide a clean dashboard layout.

Projects Tab

Viewing Projects

Projects can be viewed in grid (card) or list (table) format. Each project shows its title, status, priority, category, assigned contact, start/end dates, and completion percentage. Click a project card or row to open its detail page with all fields, associated tasks, team members, and a timeline view.

Creating and Editing Projects

To create a new project, an Admin must promote a submitted idea. Standard users can submit ideas through the "💡 Submit Idea" button. Once promoted, projects can be edited by clicking into the detail view and modifying fields. Changes save directly to the ArcGIS Online projects feature service.

Project Statuses

Projects follow this lifecycle:

Idea → Active → On Hold / Complete / Canceled

The "Future" status is also available for planned work not yet started.

Tasks Tab

Tasks are individual work items associated with projects. Each task has a title, status, assignee, due date, and parent project reference.

Creating Tasks

Click the "＋ New Task" button in the toolbar. Select the parent project, assign a team member, set a due date, and provide a description.

Task Statuses

Tasks use their own status set: Not Started, In Progress, Complete, Deferred, and Canceled.

Resources Tab

The Resources tab is the team capacity planning hub. It shows how each team member's time is allocated across projects and whether they are over- or under-utilized.

Team Member Cards

The top section, labeled "Current Week Project Allocations," displays a card for each team member showing their name, role, and current week utilization percentage. The utilization badge is color-coded:

  • Green — Under 70% utilized (has capacity)
  • Amber — 70–90% utilized (getting busy)
  • Red — Over 90% utilized (at or over capacity)

Click a card to view that person's detailed allocation below.

Allocation Chart

A 20-week stacked bar chart shows allocation data, with each project in a different color. The capacity line (dark dots) shows the person's maximum project hours per week. A subtle highlight marks the current calendar week. Use the left/right arrows to scroll through the year.

KPI Row

Below the person's name, KPI cards show: current week utilization %, project capacity hours, allocated hours, number of active projects, and year-to-date total allocated hours.

Allocation Table

A table lists each project the selected person is allocated to, showing project status, a sparkline of recent allocation, and current week allocation percentage and hours.

Editing Allocations

Click "✏️ Edit Allocations" to open the allocation editor popup. This shows a grid of projects (rows) by weeks (columns). Enter allocation percentages (0–100%) for each project and week.

Tip: Projects are sorted alphabetically. Pressing Tab moves down the column (same week, next project) rather than across a row, making it easy to fill in one week at a time.

The total row at the bottom shows combined allocation — totals over 100% are highlighted in red. Click "✓ Apply Changes" to save. Changes are written to the ArcGIS Online allocations service and the view updates immediately.

Forecast Tab

The Forecast tab projects team capacity into the future based on current allocation patterns. It shows how loaded each person is over a configurable window (default 13 weeks from the current week), helping managers identify periods of over- or under-capacity and plan resource assignments accordingly.

What the Forecast Shows

The Forecast tab has three main sections:

  • Summary Cards: One card per team member, sorted by most available. Each card shows the person's average free hours per week over the forecast window, their average utilization percentage, and a sparkline of weekly utilization.
  • Team Load Chart: An area chart showing the entire team's combined allocated hours vs. combined capacity over the forecast window. This reveals when the team as a whole is approaching capacity limits.
  • Utilization Heatmap: A detailed grid showing each person's utilization (or available hours) per week, color-coded from green (low utilization) through amber to red (at or over capacity). This makes it easy to spot who has capacity and when.

How Future Availability is Calculated

The forecast uses a "snapshot" approach to project current workloads into the future. Here's how it works step by step:

Step 1 — Past/Present Weeks: Actual Data

For all weeks up to and including the current week, the forecast uses actual entered allocation hours — no estimation. These are the real values from the allocations service, so past data is always accurate.

Step 2 — Find the Snapshot Week

For each person, the system walks backward from the current week to find the most recent week where any allocations were entered (i.e., total allocation fractions across all projects are greater than zero). This becomes the person's "snapshot week."

For example, if James has allocation data entered through March 8, then March 8 is his snapshot week — even if the current calendar week is March 15.

Why a snapshot? This approach preserves the real allocation split (all project fractions summing to roughly 100%) from a single point in time, rather than mixing stale data from different weeks where projects may have changed.

Step 3 — Project Forward

For each project that had a non-zero allocation fraction in the snapshot week, the system carries that same fraction forward into every future week. So if James was 33% allocated to a project in the snapshot week, the forecast assumes 33% continues indefinitely.

The allocated hours for each future week are calculated as:

Future Allocated Hours = Snapshot Fraction × Project Capacity for that week

Note that project capacity can vary week to week due to absences — a future vacation week will show reduced capacity (and therefore reduced allocated hours) if the absence has been entered in the Settings tab.

Step 4 — Respect Project End Dates

If a project has an end date and a future week falls after that date, the allocation drops to zero for that project. This is how the forecast shows capacity "freeing up" when projects end — revealing future availability for new work.

Step 5 — Calculate Availability

For each person, for each week:

Availability = Project Capacity − Total Allocated Hours

Where Project Capacity uses the standard formula: (40 − Absence Hours) × 0.75 × Project Time %.

Utilization is then: Total Allocated Hours ÷ Project Capacity, capped at 125% for display purposes.

Worked Example

James McGinnis is 63% project-allocated with no absences. In his snapshot week (March 8), he has:

ProjectSnapshot Fraction
Safe Cities Dashboard33%
Parking Utilization Analysis25%
Water Quality Mapping20%

For a future week with no absences, his project capacity is 18.9 hours. The forecast calculates:

ProjectCalculationHours
Safe Cities Dashboard0.33 × 18.96.24h
Parking Utilization0.25 × 18.94.73h
Water Quality Mapping0.20 × 18.93.78h
Total Allocated14.74h

Availability = 18.9 − 14.74 = 4.16 hours free. Utilization = 14.74 ÷ 18.9 = 78%.

If the Water Quality Mapping project ends in two weeks, then from that point forward James's forecast would show only 10.97h allocated and 7.93h free (58% utilization) — making him a good candidate for new work.

Limitations

Important: The forecast is a projection, not a plan. Keep these limitations in mind:
  • No allocation data = fully available: If a person has never had allocations entered (snapshot week is -1), their future shows as 100% available since there's nothing to project forward.
  • Future absences must be pre-entered: The forecast uses the capacity formula, which depends on absence data. If someone has a planned vacation next month but the absence hours haven't been entered in the Settings tab yet, the forecast won't reflect that reduced capacity.
  • Static snapshot: The forecast doesn't account for planned changes in allocation. If you know someone will move from Project A to Project B next month, the forecast won't reflect this until you actually update their allocations.
  • No new project predictions: The forecast only projects existing allocations forward — it can't predict when new projects will start consuming capacity.

Settings Tab (Admin Only)

The Settings tab is only visible to members of the Team Leads group in ArcGIS Online. It provides team management capabilities.

Team Members Table

Lists all team members with their name, role, team, skill, and project allocation percentage. From each row you can:

  • ✏️ Edit: Update the person's role, team, skill, and project time percentage. Changes save directly to the team_members REST service.
  • 📅 Absences: Opens an inline absence editor showing 8 weeks at a time, centered on the current week. Enter hours off per week (0–40). Changes auto-save to the absences REST service. A read-only "Proj Capacity" row shows the impact on available project hours.
  • 🗑 Delete: Removes the team member from the roster (with confirmation). Allocation and absence history is preserved in the database.

Adding Team Members

Click "＋ Add Member" to open a form for adding a new team member. Enter their full name, role, team, skill area, and the percentage of their work week allocated to project work.

Capacity & Utilization: Assumptions and Math

This section explains how the application calculates project capacity, allocated hours, and utilization. Understanding these calculations is important for interpreting the data on the Resources and Forecast tabs.

Core Assumptions

AssumptionValue / Explanation
Base Work Week40 hours per employee
Productivity Factor75% — Only 75% of available work hours are assumed to be productive project time. The remaining 25% accounts for meetings, email, administrative tasks, context-switching, and breaks. This is based on workplace productivity research showing the average knowledge worker is productively focused for roughly 6 of 8 hours per day.
Project Time %Each team member has a configurable percentage of their productive time dedicated to project work (e.g., 63% for a Lead Data Analyst). The remainder goes to operational duties, support, and overhead not tracked as projects.
AbsencesHours off per week (vacation, sick, holidays) are entered in the Settings tab. These reduce the base 40-hour week before the productivity factor is applied.

The Capacity Formula

For each person, for each week:

Project Capacity = (40 − Absence Hours) × 0.75 × Project Time %

Worked Example — No Absences

James McGinnis has a project time allocation of 63% and no absences this week:

StepCalculation
Base work week40 hours
Subtract absences40 − 0 = 40 hours available
Apply 75% productivity40 × 0.75 = 30 productive hours
Apply 63% project time30 × 0.63 = 18.9 project hours

Worked Example — With 1 Day Off

If James takes 8 hours (1 day) off:

StepCalculation
Base work week40 hours
Subtract absences40 − 8 = 32 hours available
Apply 75% productivity32 × 0.75 = 24 productive hours
Apply 63% project time24 × 0.63 = 15.12 project hours

Allocated Hours

When a team member is allocated a percentage to a project for a given week:

Allocated Hours = Allocation % × Project Capacity

For example, if James is allocated 50% to a project with 18.9 hours capacity: 0.50 × 18.9 = 9.45 hours. The total allocated hours for a person is the sum across all their project allocations for that week.

Utilization

Utilization measures how much of a person's available project capacity is actually allocated:

Utilization = Total Allocated Hours ÷ Project Capacity

Utilization is displayed as a percentage. 100% means fully allocated. Over 100% indicates overallocation. The color coding is:

  • Green — Under 70%
  • Amber — 70–90%
  • Red — Over 90%
Note: Both weekly_allocated and utilization are always computed client-side from the actual allocation data. This means they are always in sync with the latest edits, even before a page refresh.

Data Architecture

All data is stored in ArcGIS Online as hosted feature services. The application queries these services via the ArcGIS REST API using the authenticated user's OAuth token.

Feature Services

ServicePurposeKey Fields
projectsAnalytics project records with metadata, dates, status, and assignmentstitle, status, priority, category, contact, start, end
tasksIndividual work items linked to projectstitle, status, assignee, due_date, project_id
team_membersTeam roster with roles and capacity settingsname, role, team, skill, proj_pct
absencesPer-person, per-week time off recordsname, week_date, absence_hours
allocationsPer-person, per-project, per-week allocation recordsname, project, week_date, fraction, hours
Note: A weekly_capacity service also exists but is no longer queried by the application. All capacity values are computed dynamically from absences and team member settings.

Week Structure

The application uses a 52-week calendar year (2026) where each week starts on Sunday. Week dates are stored as Date Only fields in YYYY-MM-DD format. The week of January 4, 2026 is Week 1.

Data Flow

On page load, the application queries projects, tasks, team_members, absences, and allocations in parallel. The data is reconstructed into an in-memory structure used for all rendering. When changes are made (editing allocations, updating absences, managing team members), the application writes back to the appropriate REST service and updates local data immediately for a responsive UI.

Access Control & Permissions

Authentication

All users must authenticate through ArcGIS Online OAuth 2.0 (Implicit Grant flow, 120-minute token lifetime). Expired tokens trigger automatic re-authentication.

Group-Based Permissions

Administrative privileges (Settings tab access, Idea promotion) are controlled by membership in a designated ArcGIS Online group (Team Leads). This is managed entirely through ArcGIS Online group membership — no application-level user management is needed.

Feature Service Permissions

Read and write access to the underlying feature services is controlled through ArcGIS Online sharing settings. The following services must have editing enabled: projects, tasks, team_members, absences, and allocations.
